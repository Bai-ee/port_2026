Scout Intake · Description Wireframe

How onboarding data becomes card descriptions

This document maps the actual intake path from the worker entrypoint, through evidence capture, synthesis, analyzer aggregation, normalization, persistence, and frontend rendering. It also shows which files influence each onboarding card description, where the deterministic description system takes over, and where to adjust logic when communication drifts away from the facts.

Purpose

  • Trace which source files shape card copy.
  • Separate raw evidence problems from description-layer problems.
  • Identify where to tweak accuracy without guessing.
  • Capture review notes directly in the document.

Key Description Rule

  • One dominant signal per card.
  • Express it as: what we saw → why it matters → what to do next.
  • Prefer deterministic selectors for tile/modal alignment.
  • Use richer synthesis only when the fact pack is too thin or ambiguous.

Pipeline Wireframe

This is the live processing chain for onboarding and the exact places where content can become more or less accurate. Each step lists the main files involved and what they contribute downstream.

1

Run trigger and worker handoff

The worker route claims the queued run and hands the client config into the intake orchestrator. This is where a stale run or the wrong website URL can poison every downstream result.

app/api/worker/run-brief/route.js api/_lib/run-lifecycle.cjs
Inputs: clientId, clientConfig, sourceInputs.websiteUrl Failure mode: wrong target URL or stale run state
2

Intake orchestration

The intake runner coordinates every downstream stage: site fetch, synthesis, analyzers, card skills, PageSpeed, scout-config generation, screenshot artifacts, and the SEO guardian. If a stage is skipped, timed out, or downgraded to a warning, that fact starts here.

features/scout-intake/runner.js

What it does

  • Normalizes the website URL.
  • Runs evidence capture and synthesis in sequence.
  • Collects warnings, artifacts, analyzer outputs, and scribe output.
  • Calls the SEO guardian before the result is normalized and persisted.

What to tweak here

  • Timeouts and retry behavior.
  • Whether missing evidence downgrades to partial or critical.
  • Which artifacts are considered required for a given card.
  • Which warning codes should appear in the debug surfaces.
3

Evidence capture and artifact generation

These modules gather the raw facts. If copy is wrong, the first question is whether the underlying evidence was wrong, thin, or missing before it ever reached a selector.

features/scout-intake/site-fetcher.js features/intelligence/pagespeed.js features/scout-intake/design-system-extractor.js api/_lib/browserless.cjs api/_lib/device-mockup.cjs features/scout-intake/scout-config-generator.js

Raw outputs

  • HTML, metadata, page evidence, and crawl findings.
  • PageSpeed scores, CWV, warnings, host/service/provider hints.
  • Screenshot and device mockup artifacts.
  • Extracted colors, typography, and design system signals.
  • Scout config used by the data map/admin preview surfaces.

Primary accuracy levers

  • Improve service/provider detection.
  • Separate measurement limitations from real site issues.
  • Expose more structured diagnostics instead of raw error strings.
  • Route all useful evidence into rawData before description building.
4

Interpretation, analyzers, and synthesis

The pipeline converts raw evidence into higher-level findings, gaps, strengths, readiness states, and recommendation context. This is where generalized language can creep in if the outputs are not tied tightly enough to the fact pack.

features/scout-intake/intake-synthesizer.js features/scout-intake/analyzers/* features/scout-intake/skills/_runner.js features/scout-intake/skills/_aggregator.js features/scout-intake/scribe.js features/scout-intake/seo-comment-guardian.js

Interpretation layers

  • Synthesizer: broad site understanding and early narrative context.
  • Analyzers: deterministic/domain-specific findings and readiness.
  • Skills: card-specific enrichment and aggregation.
  • Scribe: fallback expanded copy and brief content.
  • SEO guardian: fact-locked SEO writer/reviewer using the canonical fact pack.

Primary accuracy levers

  • Prevent stale or contradictory findings from surviving into aggregate.
  • Filter measurement-failure language when PageSpeed actually succeeded.
  • Keep AI-specific findings precise, not generic.
  • Constrain synthesis to facts instead of letting it invent status claims.
5

Normalization and persistence

The normalized result becomes the stable write shape for Firestore and the dashboard projection. If a useful fact exists upstream but never appears in the UI, this layer is often the missing bridge.

features/scout-intake/normalize.js api/_lib/run-lifecycle.cjs
Writes: brief_runs, clients/{clientId}/brief_runs, dashboard_state/{clientId} Projection: snapshot, signals, strategy, siteMeta, analyzerOutputs, scribe, artifacts
6

Frontend card rendering and debug surfaces

The dashboard assembles rawData per card, resolves analyzer aliases, runs the deterministic selector, and decides whether built copy, guardian copy, or fallback copy should render. The admin surfaces expose the live card inputs so you can inspect what actually reached the UI.

DashboardPage.jsx features/scout-intake/card-description-builder.js app/api/admin/scout-card-copy/route.js app/preview/scout-config/page.jsx

Critical render facts

  • resolveAnalyzerSource() maps display cards to analyzer source ids.
  • buildCardDescription() selects one dominant signal.
  • Deterministic cards ignore scribe.short in the tile path.
  • The admin copy route shows rendered copy alongside raw card inputs.

Primary accuracy levers

  • Improve per-card rawData mapping.
  • Refine dominant-signal selectors and copy templates.
  • Sanitize internal tokens before user-facing render.
  • Expose richer debug data in modal/data tabs when review is blocked.

Description Development Flow

This is the exact description path to keep in mind when a card feels wrong. The tile should not be treated as a generic summary layer; it is a deterministic decision system that sits on top of normalized evidence.

Input assembly

DashboardPage.jsx assembles rawData per card from live projection data such as PageSpeed, AI visibility, mockups, social metadata, industry, style-guide, or priority data.

Analyzer resolution

card-description-builder.js resolves the canonical analyzer aggregate using resolveAnalyzerSource() so alias logic does not live inline in the dashboard.

Dominant signal selection

The selector for that card chooses one signal only, then writes: finding → impact → action. This is the core pattern that keeps cards readable.

SEO special handling

seo-comment-guardian.js builds a fact pack, filters contradictory analyzer findings, then generates aligned short and expanded SEO copy from the same canonical signal path.

Fallback behavior

If a deterministic selector has no usable signal, the UI may fall back to static copy or older scribe paths. That is usually the first branch to audit when a card suddenly feels generic.

Admin validation path

app/api/admin/scout-card-copy/route.js exposes card inputs and rendered copy so the data-map view can show exactly what the selector had access to at render time.

Accuracy rule of thumb: If the shell values are right but the description is wrong, the problem is usually in rawData mapping, dominant-signal selection, or fallback rendering. If both shell and description are wrong, start upstream in evidence capture or analyzer aggregation.

Review Notes

Use these fields to note where communication is drifting and what should change. Notes are stored in this browser using localStorage.

Best place to tweak evidence quality Evidence modules and warning routing in runner.js, site-fetcher.js, and pagespeed.js.
Best place to tweak what the user sees DashboardPage.jsx, card-description-builder.js, and seo-comment-guardian.js.
Best place to verify the final inputs app/api/admin/scout-card-copy/route.js and /preview/scout-config.

High-Value Tweak Zones

These are the highest-leverage places to improve accuracy before changing architecture.

1. Raw evidence completeness Improve capture for performance diagnostics, provider hints, social metadata, schema types, and device artifacts before changing wording.
2. rawData mapping in DashboardPage Make sure each card receives the facts it actually needs. Many communication bugs are simply missing or misrouted raw inputs.
3. Dominant signal hierarchy Tune buildCardDescription() so the most distinctive issue or strength wins, not just the first available issue.
4. Guardrails against contradiction Keep the SEO guardian fact-locked and extend that pattern to other high-risk cards if needed.
5. Surface-level sanitization Strip internal token ids, raw font keys, or implementation artifacts before they reach the UI.
